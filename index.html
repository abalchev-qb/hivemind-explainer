The why is invisible at HEAD

Code tells you what.
It rarely tells you why.

Open any file at HEAD and you can read exactly what it does. What you can't see is the rejected alternative, the vendor quirk, the reviewer's objection, the bug that forced the shape — the WHY. It lived in a PR description, a Jira thread, or a reviewer comment, and then it evaporated.

Hivemind walks your project's history, extracts those non-obvious WHYs as small durable lessons, and serves them back the moment someone — a person or an agent — needs them.

In one line: a shared, searchable memory of why the code is the way it is — grown automatically, read on demand.

Phase 1 · enumerate

It starts at a folder and walks outward

In plain terms: follow the trail of history around some code — every commit, the PR that merged it, the ticket behind that PR — collecting everything that might carry a WHY.

You run /learn-from <folder> and point Hivemind at an origin — a feature folder like features/billing-dashboard. A deterministic crawler runs a breadth-first walk over the graph of related artifacts: the commits that touched the folder, the PRs that merged them, the Jira tickets they reference, and the files they changed. No LLM here — just gh, git, and acli, so this step is cheap and repeatable.

Phase 2 · classify

A fast model votes on every node

In plain terms: a cheap model triages each artifact into one of three buckets — teach it, ignore it, or this is a bug, find what caused it.

Each node is handed to a cheap Haiku classifier in batches. One question dominates: “could an engineer fluent at HEAD re-derive the WHY without history?” If yes → skip. If a hidden WHY is present → learn. If it's a bug-fix or revert → backtrack (the fix proves a past decision was wrong).

learn → write a lesson backtrack → find the bug's origin skip → re-derivable, drop it

Under-emission is the safe failure mode — Hivemind is a lesson book for experts, not a changelog. On a real billing-dashboard run: 208 learn, 211 backtrack, the rest skipped.

Obvious-skip nodes (empty-description tickets, pure file breadcrumbs) are filtered in Python first, so they never cost a token.

Phase 3 · extract

Lessons get written — sized to the signal

A learner (Sonnet, or Opus for the hard ones) turns each learn verdict into one tiered lesson. For each backtrack, a backtracker first walks from the fix back to the introducing commit, then hands the learner a counterfactual: “at introduction, what should have been done to prevent this bug?”

Phase 4 · consolidate & publish

Dedup, then make it searchable

In plain terms: merge the lessons that say the same thing, then file the survivors in a vector database so they can be found by meaning, not just keywords.

Many learners fire in parallel, so a run can produce near-duplicates. A consolidator clusters lessons by shared evidence (same source, lens, scope path, or PR) and merges or drops the redundant ones. Survivors are POSTed to the storage API → written to S3 → embedded into a Bedrock Knowledge Base backed by Aurora pgvector.

  • Identity, not content. A lesson can gain evidence and grow T1→T2 across runs while keeping the same id.
  • Idempotent. Re-running on the same folder upserts instead of piling on near-duplicates.
  • Auditable. Every run also uploads a full event log — what was visited, classified, learned, and refused.
The payoff · retrieve

The WHY shows up exactly when it's needed

In plain terms: ask a plain-English question, get back the handful of past lessons that matter — so you read the scar before you re-open the wound.

Months later a different engineer touches the same area. Before they write code, they (or their agent) ask the hivemind. A Haiku retriever extracts keywords, does a semantic search over the KB, self-checks, and returns up to 5 lessons — the calling agent only ever sees the compact result, not the search work.

The autonomous path · forage

It also learns while you work — no command needed

In plain terms: the best WHY is usually spoken, not committed. When you correct an agent — “no, don't do that, we tried it and it broke X” — Hivemind quietly catches that and turns it into a lesson.

The graph walk is retrospective archaeology — it mines WHYs that survived in commits and PRs. But most reasoning never reaches an artifact; it lives in the moment a human pushes back on an agent. The forager captures exactly that signal, passively, at near-zero cost.

1 · Pre-filter

Regex, no LLM, <50ms. Runs on every prompt. Catches correction-shaped language — “no”, “actually”, “wait”, “that's wrong”. Rejects ~90% instantly. Zero user-visible latency.

→

2 · Pollen

Haiku draft, ~$0.0001. In the background, a cheap model captures the moment as loose “pollen”: what was corrected, the stated WHY, the agent's prior position. Local only — never shipped yet.

→

3 · Promote

Opus at session end, ~$0.15. Reads the full transcript + all pollen. Promotes real lessons into the hive, demotes the ones that were just the agent explaining itself, discards noise.

why two tiers (cheap draft → smart promote)
  • Catch fast, judge slow. Haiku grabs the moment before it's gone; Opus decides later with the whole conversation in view.
  • Not every correction is a lesson. If you pushed back and then accepted the agent's explanation, that's a “teach the agent to explain better” signal — not a corpus lesson. Only Opus-with-full-context can tell them apart.

The payoff for agents · the flywheel

An autonomous agent that gets wiser every run

In plain terms: an autonomous agent normally starts every task with amnesia. Wire in Hivemind and it walks in already knowing the project's scars — and it leaves the project a little smarter than it found it.

Agents like qbee (and our self-verifying agent, in progress) turn a Claude session into a worker that branches, codes, commits, and opens a PR on its own. Hivemind gives that worker memory across runs through three touch-points:

BEFORE

Query, then code

Before touching an unfamiliar area, the agent calls hivemind:query. The hidden constraints, rejected approaches and past bugs land in its context — so it doesn't re-walk into them.

DURING

Self-verify against the hive

A self-verifying agent re-queries before it declares done: “does my change violate anything the hive already knows?” The forager, meanwhile, captures any course-correction along the way.

AFTER

Learn from what it shipped

Once the agent opens its PR, /learn-from <that PR> mines durable lessons from the work it just produced — feeding the next agent's before step.

the full loop

Walk → classify → learn → store → retrieve

Two ways in, one growing memory: the /learn-from walk mines history on demand; the forager catches live corrections for free. Both feed the same store — read back by people and agents via hivemind:query.
